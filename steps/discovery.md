## Objective

Understand what the project is before anyone touches technology. Build a clear picture of vision, users, problem, and success criteria through conversation — not a form.

## Process

1. **Open with the big picture.**
   Ask: "What's this project for? Give me the elevator pitch."
   Listen. Follow up on whatever is unclear or missing.

2. **Identify the users.**
   Ask: "Who needs this?" Follow threads: primary users, secondary users, admin/ops roles.
   One question at a time. Adapt based on what the user volunteers.

3. **Nail down the problem.**
   Ask: "What problem does this solve? What happens today without it?"
   Push past solution descriptions to the underlying need.

4. **Define success.**
   Ask: "What does success look like? How would you know this worked?"
   Get concrete criteria, not aspirations. Numbers beat adjectives.

5. **Synthesize into a project brief.**
   Combine everything into a structured brief. Present it back for confirmation.

## User Interaction

- One question at a time. This is a conversation between peers, not a questionnaire.
- If the user gives a rich answer that covers multiple areas, skip the questions already answered.
- Push back on vague answers. "Make it easy to use" is not a success criterion.
- State your understanding and let the user correct it.

## Outputs

- Project brief containing:
  - **Vision**: one-sentence description of what this is
  - **Users**: who needs it and why
  - **Problem**: what exists today and why it is insufficient
  - **Success criteria**: measurable outcomes that define done

## Completion Criteria

- Vision, users, problem, and success criteria are all documented.
- The user has confirmed the brief reflects their intent.
- Enough context to move into research or specification.
